# Send 和 Sync — 编译期并发证明 🟠

> **你将学到：** Rust 的 `Send` 和 `Sync` 自动 trait 如何将编译器转化为并发审计引擎 —— 在编译期证明哪些类型可以跨越线程边界、哪些可以安全地被共享，且运行时开销为零。

## 问题：没有安全网的并发访问

在系统编程中，外设寄存器、共享缓冲区和全局状态通常会从多个上下文（如主循环、中断处理程序 ISR 和 DMA 回调）中被并发访问。在 C 语言中，编译器无法强制执行并发安全，开发者只能依靠文档或手动加锁来避免数据竞争。

## Send 和 Sync 的证明作用

Rust 定义了两个“标记 trait” (marker traits)，编译器会自动推导出这些属性：

| Trait | 证明含义 | 类型定义 |
|-------|-------|-------------------|
| `Send` | 该类型的值可以被安全地 **转移 (Move)** 到另一个线程 | “允许跨越线程边界” |
| `Sync` | 该类型的 **共享引用 `&T`** 可以由多个线程安全使用 | “允许多线程读取共享” |

这些是 **自动 Trait (Auto-traits)** —— 编译器通过检查结构体中的每一个字段来自动推导出它们。如果结构体包含任意 `!Send` 字段（如 `Rc<T>` 或原始指针 `*const T`），那么整个结构体就会变成 `!Send`。

## 线程封闭的硬件句柄

在嵌入式系统中，特定的硬件寄存器块通常只能从单个执行上下文中访问。由于原始指针天生就是 `!Send` 和 `!Sync` 的，将它们封装在结构体中会自动实现线程限制：

```rust
pub struct Uart {
    regs: *const u32, // 包含原始指针，自动变为 !Send + !Sync
}

// 尝试跨线程发送 `Uart` 句柄将在编译时失败。
```

## 创建安全的共享状态

- **`Mutex<T>`**：通过强制所有访问都必须经过一个锁，将 `!Sync` 类型转化为 `Sync` 类型。编译器会保证：除非持有锁，否则你无法获取内部数据的引用。
- **`Arc<T>`**：允许在多线程间共享 `Send + Sync` 类型数据的所有权。

## 关键收获

1. **编译器是并发审计师** —— `Send` 和 `Sync` 是自动推导的。你不会因为疏漏而忘记标记一个非线程安全的类型。
2. **零运行时成本** —— 这些 trait 仅存在于类型系统中，编译后完全消失。
3. **硬件隔离** —— 利用 `!Send` 句柄将硬件访问锚定在特定的核心或线程上，防止中断冲突。
4. **无惧并发 (Fearless Concurrency)** —— 结合所有权和 trait 约束，Rust 将原本动态的并发检查转化为了静态的、结构上的证明。
